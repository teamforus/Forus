# Webshop translations

How text is translated on the websites, and how to change it safely. Please be aware that the Dutch language is leading, the other languages are machine translated and are not requiered for a PR. In most cases, we actually prefer you to just add the Dutch texts and we will conduct the translations during the PR review. So, only use this for testing or development learning purposes.

## TL;DR

- Dutch (`nl`) is the source of truth.
- All other languages are generated by DeepL via scripts in `forus-frontend-react/translations/`.
- Generation of the other languages is optional.
- Edit **only** the Dutch `.mjs` source files. Regenerate everything else with one command.
- Never hand-edit `translations/cache/cache.json` or `react/src/webshop/i18n/translated/*.json`.

## Scope

These scripts translate the **webshop** only (`react/src/webshop/i18n/...`). The dashboard has its Dutch strings hardcoded in components and services; changing those today is manual per file.

## File layout (in `forus-frontend-react/`)

- Dutch source (hand-edited):
  - `react/src/webshop/i18n/i18n-nl.mjs` — aggregator, imports every namespace.
  - `react/src/webshop/i18n/nl/**/*.mjs` — one file per page / component / modal.
- Generated per-language output (do not edit):
  - `react/src/webshop/i18n/translated/{ar,de,en-US,fr,pl,ru,tr,uk}.json`
- Diff anchor / cache (do not edit):
  - `translations/cache/cache.json` — last snapshot of the Dutch source.
- Scripts:
  - `translations/prepare.mjs` — diff current Dutch vs cache, writes `cache_added.json` + `cache_removed.json`.
  - `translations/translate.mjs` — calls DeepL for each target language, writes `cache_translated_<lang>.json`.
  - `translations/apply.mjs` — merges new translations into `translated/*.json` and removes deleted keys.
  - `translations/update-cache.mjs` — updates `cache.json` to match current Dutch.
  - `translations/helpers.mjs` — shared helpers, loads `deepl.env.mjs`.

## DeepL configuration (`deepl.env.mjs`)

The scripts require a DeepL API key. It lives at the submodule root:

```
forus-frontend-react/deepl.env.mjs
```

`deepl.env.mjs` and `deepl.env.*.mjs` are gitignored — only the committed `deepl.env.example.mjs` template is tracked. **Never commit your key.**

Setup:

```bash
cd forus-frontend-react
cp deepl.env.example.mjs deepl.env.mjs
```

Then edit `deepl.env.mjs`:

```javascript
export default {
    testMode: false,
    accessKey: 'YOUR-DEEPL-KEY',
    sourceLanguage: 'nl',
    targetLanguages: ['ar', 'de', 'en-US', 'fr', 'pl', 'ru', 'tr', 'uk'],
};
```

- `testMode: true` skips DeepL and prefixes each string with the target lang code — useful for a dry run.
- Open-source contributors: use your own free or paid DeepL key; keys are not shared across the project.
- Keep `targetLanguages` in sync with the files in `react/src/webshop/i18n/translated/` or those languages won't be updated.

## Changing a string

1. Edit the relevant Dutch source file, e.g. `react/src/webshop/i18n/nl/pages/product.mjs`.
2. From `forus-frontend-react/`, run:

   ```bash
   npm run translations-run-all
   ```

   Defined in `package.json` as:

   ```
   translations-prepare → translations-translate → translations-apply → translations-update-cache
   ```

3. Commit together:
   - Your Dutch source changes.
   - The updated `react/src/webshop/i18n/translated/*.json`.
   - The updated `translations/cache/cache.json`.

## Adding a new target language

1. Add the language code to `targetLanguages` in `deepl.env.mjs`.
2. Run `npm run translations-run-all`. A new `translated/<lang>.json` appears.
3. Wire the language into the webshop i18n loader if it's a new UI locale.

## Common mistakes

- **Hand-editing `translated/*.json` or `cache.json`.** These get overwritten / drift from the cache; the next run either re-translates everything or silently misses changes.
- **Running only part of the chain.** Prefer `translations-run-all` unless you know exactly which step you're skipping.
- **Forgetting to commit the regenerated files.** The PR should include the Dutch source changes *and* the regenerated outputs, or CI / other devs will see stale translations.
- **Using variables without `{{ }}`.** Variables wrapped in `{{ }}` are auto-protected (the script wraps them in `<keep>` tags for DeepL). Anything else is treated as translatable text.

## Troubleshooting

- **DeepL auth errors** → check `accessKey` in `deepl.env.mjs` and the plan quota.
- **Partial output / empty strings** → rerun with `testMode: true` to confirm the input pipeline is correct, then flip back to `false`.
- **Unexpected diff in `cache.json`** → usually means someone edited a generated file by hand. Regenerate from a clean Dutch source.
